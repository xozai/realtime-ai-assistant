# Realtime AI Assistant

[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](https://opensource.org/licenses/MIT)

A Python 3.11+ realtime AI assistant for software discovery calls. It opens an OpenAI Realtime API WebSocket session, guides a requirements conversation, captures requirements through tool calls, and generates Agile user stories with Pydantic structured output.

Inspired by [`disler/poc-realtime-ai-assistant`](https://github.com/disler/poc-realtime-ai-assistant) — reuses its async Realtime API event loop, tool-chaining pattern, Rich terminal logs, in-session memory, and structured LLM parsing.

## Detailed Installation and Configuration

Use this section when setting up the app on a new machine or preparing a clean local environment for a discovery session.

### Setup prompt for another operator or AI assistant

Use this prompt when you want another operator or AI assistant to install, configure, and smoke-test the app from scratch:

```text
Install and configure the Realtime AI Assistant from the xozai/realtime-ai-assistant repository. Use Python 3.11+, create an isolated virtual environment, install dependencies from requirements.txt, copy .env.example to .env, configure OPENAI_API_KEY, optionally configure STORY_GENERATION_MODEL and Jira settings, then run the CI-equivalent checks. After setup, start the app in text mode, open the dashboard, capture sample requirements, generate user stories, refine one story, analyze coverage, export stories, and verify the dashboard updates without manual refresh. Do not commit secrets. Document any environment-specific setup issues and their fixes.
```

### 1. Prerequisites

Install or verify these tools first:

- **Python 3.11 or newer** — the app is developed and tested on Python 3.11+.
- **Git** — required to clone the repository.
- **OpenAI API key** — required for the Realtime API, Chat Completions story generation, confidence scoring, and embeddings.
- **Optional microphone/speaker support** — required only for `--voice`; install the `sounddevice` Python package and make sure PortAudio-compatible audio devices are available.
- **Optional Jira credentials** — required only when creating Jira issues; dry-run Jira previews can be used to validate payloads before creating anything.

Check Python before installing:

```bash
python3 --version
```

If your system has multiple Python versions, use the explicit Python 3.11+ binary in every command below, for example `python3.11`.

### 2. Clone and enter the repository

```bash
git clone https://github.com/xozai/realtime-ai-assistant.git
cd realtime-ai-assistant
```

If you already have the repository, update it first:

```bash
git checkout main
git pull --ff-only origin main
```

### 3. Create and activate a virtual environment

```bash
python3.11 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
```

On Windows PowerShell, activate with:

```powershell
.venv\Scripts\Activate.ps1
```

### 4. Install dependencies

```bash
pip install -r requirements.txt
```

For voice mode, also install audio support:

```bash
pip install sounddevice
```

### 5. Create `.env`

```bash
cp .env.example .env
```

Edit `.env` and set at least `OPENAI_API_KEY`:

```bash
OPENAI_API_KEY=sk-your-key-here
```

Do not commit `.env`. It contains local secrets.

### 6. Configure story generation

Story generation uses Chat Completions separately from the Realtime conversation model. Configure it with either `.env` or the CLI.

In `.env`:

```bash
STORY_GENERATION_MODEL=gpt-4o
```

Or per run:

```bash
python src/realtime_assistant/main.py --story-model gpt-4o-mini
```

Precedence is:

1. `--story-model`
2. `STORY_GENERATION_MODEL`
3. built-in default `gpt-4o`

### 7. Configure Jira integration, optional

Add Jira settings only if you want the assistant to create Jira Story issues:

```bash
JIRA_BASE_URL=https://your-org.atlassian.net
JIRA_USER_EMAIL=you@example.com
JIRA_API_TOKEN=your_jira_api_token_here
JIRA_STORY_POINTS_FIELD=story_points
```

Notes:

- `JIRA_BASE_URL` should be the Atlassian site root, not a project URL.
- `JIRA_USER_EMAIL` is the email associated with the Atlassian API token.
- `JIRA_API_TOKEN` should be an Atlassian API token, not your account password.
- `JIRA_STORY_POINTS_FIELD` is optional and depends on your Jira field configuration.
- The discovery `--project` key is not the same as the Jira project key. Pass the Jira project key to `submit_stories_to_jira` or `POST /api/jira/{project_key}`.

Before creating real Jira issues, use dry-run mode:

```text
submit_stories_to_jira(project_key="PROJ", dry_run=true)
```

or:

```bash
curl -X POST 'http://localhost:8000/api/jira/PROJ?dry_run=true'
```

### 8. Configure deduplication and cost rates, optional

Requirement deduplication uses embeddings. The default similarity threshold is `0.85`. Lower values merge more aggressively; higher values keep more near-duplicates.

```bash
DEDUP_THRESHOLD=0.85
```

Token and cost tracking uses configurable per-1K-token prices:

```bash
REALTIME_INPUT_PRICE_PER_1K=0.005
REALTIME_OUTPUT_PRICE_PER_1K=0.02
CHAT_INPUT_PRICE_PER_1K=0.0025
CHAT_OUTPUT_PRICE_PER_1K=0.01
EMBEDDING_PRICE_PER_1K=0.00002
```

Update these if your OpenAI pricing or selected models differ from the defaults.

### 9. Verify the installation

Run the checks used in CI:

```bash
pip install pytest pytest-asyncio fastapi httpx uvicorn
python -m pytest tests/ -q --tb=short
```

If you use `uv`, the equivalent is:

```bash
uv sync --extra dev
uv run ruff check src/ tests/
uv run pytest tests/ -q --tb=short
```

### 10. Start the app

Default text mode with the dashboard enabled:

```bash
python src/realtime_assistant/main.py
```

The terminal will show:

- the generated session ID
- the transcript path
- the dashboard URL, usually `http://localhost:8000`
- the text prompt for entering discovery conversation messages

Type `quit` or `exit` to end the session. The app saves the session, transcript, exports, and cost summary on exit.

## Detailed Feature Usage Guide

This section walks through the main user workflows after the app is installed and configured.

### Text-mode discovery session

Start a standard discovery session:

```bash
python src/realtime_assistant/main.py
```

Use natural language to describe the product, users, constraints, and desired outcomes. The assistant can call tools during the conversation to capture requirements and later generate structured user stories.

Good prompts during a session:

```text
We are building a scheduling assistant for field technicians.
Users need to see jobs by region and receive urgent dispatch updates.
Capture that as a requirement.
Generate user stories from the requirements we have so far.
Analyze coverage.
Export the stories to Markdown and JSON.
```

End the session with:

```text
quit
```

### Scripted discovery session

For repeatable demos or smoke tests, pass prompts separated by `|`:

```bash
python src/realtime_assistant/main.py \
  --prompts "We need a task manager|Users need tasks and reminders|Generate stories|Analyze coverage"
```

The app sends each prompt in order, then saves the session as usual.

### Voice-mode discovery session

Install voice support once:

```bash
pip install sounddevice
```

Then run:

```bash
python src/realtime_assistant/main.py --voice
```

Voice mode streams microphone audio to the Realtime API and plays assistant audio responses through the local speaker. Press `Ctrl+C` to end the call and save the session.

If audio devices are unavailable or permissions are denied, use text mode instead.

### Dashboard workflow

The dashboard starts automatically unless `--no-dashboard` is passed.

Open:

```text
http://localhost:8000
```

Use the dashboard to:

- monitor captured requirements and confidence badges
- review generated user stories
- edit requirement or story fields inline
- refine a single story with targeted feedback
- view requirement coverage and low-confidence items
- generate an executive summary
- export stories to JSON and Markdown
- preview or submit Jira stories
- watch realtime updates over Server-Sent Events (`GET /api/events`)

The dashboard uses `EventSource` for live updates and falls back to polling if the SSE connection is unavailable.

### Capturing requirements

Requirements are normally captured by the assistant during conversation through:

```text
capture_requirement(requirement, category)
```

Supported categories:

- `functional`
- `non-functional`
- `constraint`
- `assumption`

Each captured requirement is:

- assigned an ID such as `REQ-4A12C9E0`
- scored for confidence (`high`, `medium`, `low`)
- checked against existing requirements for embedding-based duplicates
- displayed in the dashboard
- included in session persistence, transcripts, coverage analysis, and exports

Low-confidence requirements are intentionally surfaced so the assistant can ask clarifying questions before story generation.

### Generating user stories

Ask the assistant to generate stories after requirements have been captured:

```text
Generate user stories.
```

The app calls `generate_user_stories()` and produces Pydantic-validated stories with:

- story ID
- title
- user-story text
- acceptance criteria
- priority
- Fibonacci story points
- `source_requirement_ids` for traceability

The story-generation model is controlled by `--story-model` or `STORY_GENERATION_MODEL`.

### Refining a single story

Use selective refinement when one story needs edits but the rest of the backlog should stay unchanged.

Example natural-language prompt:

```text
Refine US-002 to focus on mobile technicians and include an offline acceptance criterion.
```

Or from the dashboard, use the per-story refine action with feedback and optional requirement IDs.

The app calls:

```text
refine_user_story(story_id, feedback, requirement_ids)
```

Selective refinement:

- preserves unrelated stories
- keeps the target story ID stable
- can use specific requirement IDs as context
- records refinement history on the session
- publishes a dashboard update event

### Requirement coverage analysis

Ask for coverage after generating stories:

```text
Analyze coverage.
```

The app calls `analyze_story_coverage()` and reports:

- overall coverage percentage
- covered and uncovered requirements
- low-confidence requirement count
- must-have gaps before Jira submission

Coverage is also available from the dashboard endpoint:

```bash
curl http://localhost:8000/api/coverage
```

### Executive summary

Generate a product/discovery summary with:

```text
Generate an executive summary.
```

The app stores the summary in the session and exposes it in the dashboard. It includes:

- overview
- key requirements
- open questions
- risks

### Exporting stories

Ask the assistant:

```text
Export the user stories.
```

The app writes JSON and Markdown exports. By default, files are created under:

```text
exports/<project_key>/
```

Customize export location and basename:

```bash
python src/realtime_assistant/main.py \
  --project acme \
  --output-dir exports/acme-demo \
  --export-name sprint_discovery
```

Exports include stories, acceptance criteria, requirement traceability, confidence information, and coverage analysis.

### Jira dry-run preview and submission

Preview Jira payloads first:

```text
submit_stories_to_jira(project_key="PROJ", dry_run=true)
```

Or through the dashboard/API:

```bash
curl -X POST 'http://localhost:8000/api/jira/PROJ?dry_run=true'
```

When the preview looks correct, create real Jira issues:

```text
submit_stories_to_jira(project_key="PROJ", dry_run=false)
```

Real submissions return per-story results. If one story fails, later stories are still attempted. Results can include:

- `success`
- `failure`
- `skipped`
- created Jira keys and URLs
- error messages for failed stories

### Session persistence and resume

Every session is saved under:

```text
sessions/<project_key>/<session_id>.json
```

Resume a previous session:

```bash
python src/realtime_assistant/main.py --project acme --resume DISC-001
```

Start a new session with a known ID:

```bash
python src/realtime_assistant/main.py --project acme --session-id DISC-042
```

`--resume` and `--session-id` cannot be used together.

### Multi-product usage

Use project keys to isolate independent products, clients, or discovery efforts:

```bash
python src/realtime_assistant/main.py --project acme-crm
python src/realtime_assistant/main.py --project beta-payments
```

Project isolation affects:

- session files
- transcript files
- exports
- dashboard project metadata
- resume behavior

### Conversation transcripts

Transcripts are enabled by default and written to:

```text
transcripts/<project_key>/<session_id>/
```

Each session creates:

- `transcript.md` — readable transcript with speaker labels
- `transcript.json` — structured transcript with timestamps and metadata

Disable transcripts when needed:

```bash
python src/realtime_assistant/main.py --no-transcript
```

### Realtime WebSocket reconnection

If the OpenAI Realtime WebSocket drops, the app retries with bounded exponential backoff and restores captured context.

Tune reconnect behavior:

```bash
python src/realtime_assistant/main.py \
  --reconnect-attempts 5 \
  --reconnect-delay 1 \
  --reconnect-max-delay 10
```

### Dashboard API quick reference

Examples while the app is running:


### Realtime dashboard updates

The dashboard opens an `EventSource` connection to `GET /api/events`. Tool handlers publish compact events whenever session state changes, including requirement capture, story generation, selective story refinement, exports, and Jira submissions. Each event includes a session snapshot so the browser can refresh counts immediately without waiting for the next polling interval.

If the SSE connection is unavailable or reconnecting, the dashboard keeps the existing REST polling fallback so the UI continues to update in constrained browser or proxy environments.

## Tool Chain

The Realtime session registers these tools with the model:

| Tool | Description |
|---|---|
| `capture_requirement(requirement, category)` | Store a requirement; auto-deduplicates via embeddings and scores confidence via LLM |
| `ask_clarifying_question(topic, question)` | Log a clarifying question |
| `summarize_requirements()` | Print requirements grouped by confidence; Rich-warns on low-confidence items |
| `generate_user_stories()` | Produce a full replacement set of structured `UserStory` objects with `source_requirement_ids` via Chat Completions |
| `refine_user_story(story_id, feedback, requirement_ids)` | Refine or regenerate one existing story while preserving unrelated generated stories and recording refinement history |
| `generate_session_summary()` | Generate a structured executive summary (overview, key requirements, open questions, risks) |
| `analyze_story_coverage()` | Compute per-requirement coverage from story source IDs; flags gaps; warns on low-confidence |
| `dedupe_requirements()` | On-demand pairwise similarity pass over stored requirements; reports near-duplicate pairs |
| `export_user_stories(format, output_dir, export_name)` | Write JSON and Markdown exports including coverage report |
| `submit_stories_to_jira(project_key, dry_run)` | Preview or create Jira Story issues; reports per-story success/failure/skipped results and warns if uncovered must-have requirements exist |

**Requirement categories:** `functional` · `non-functional` · `constraint` · `assumption`

**Requirement confidence:** `high` · `medium` · `low` — scored by LLM on capture; low items flagged for follow-up

**Story priorities:** `must-have` · `should-have` · `could-have` · `wont-have`

**Story points:** Fibonacci — `1, 2, 3, 5, 8, 13` (Pydantic-validated)

## Confidence Scoring

Every captured requirement is automatically scored by the LLM for how clearly it was articulated:

- **high** — well-defined, specific, testable
- **medium** — reasonable but could use clarification
- **low** — vague or ambiguous; the assistant is prompted to re-ask

Low-confidence items are highlighted in `summarize_requirements`, shown with a colored badge in the dashboard (green / yellow / red), included in exports, and surfaced in the coverage report's `low_confidence_count`.

---

## Requirement Deduplication

When a new requirement is captured, its embedding (`text-embedding-3-small`) is compared against all stored requirements. If cosine similarity exceeds the threshold (default `0.85`, env-configurable via `DEDUP_THRESHOLD`), the duplicate is silently merged rather than stored. A tool result explains the merge with the existing requirement ID.

Run `dedupe_requirements` at any time for an on-demand pairwise report of near-duplicate pairs still in the session.

## Token and Cost Tracking

Token usage is accumulated across the Realtime API and Chat Completions calls and stored on `DiscoverySession.costs`. At session end, a Rich table is printed:

```
┌─────────────────┬───────────────┬────────────────┬───────────────┐
│ Model           │ Input tokens  │ Output tokens  │ Est. cost USD │
├─────────────────┼───────────────┼────────────────┼───────────────┤
│ Realtime        │ 12,450        │ 3,820          │ $0.139        │
│ Chat completions│ 1,200         │ 680            │ $0.010        │
│ Embeddings      │ 340           │ —              │ $0.000        │
├─────────────────┼───────────────┼────────────────┼───────────────┤
│ Total           │               │                │ $0.149        │
└─────────────────┴───────────────┴────────────────┴───────────────┘
```

Cost rates are configurable via env vars (see Install section). The dashboard `GET /api/session` endpoint includes the full cost breakdown.

## Jira Integration

Submits generated stories to Jira as Story issues via the Atlassian REST API v3. Uses stdlib `urllib` — no extra dependencies.

Add to `.env` (see Install section). The `submit_stories_to_jira` tool requires an explicit Jira project key — it is intentionally separate from the discovery `--project` key.

Use `dry_run=true` to preview the exact Jira issue payloads before anything is created:

```text
submit_stories_to_jira(project_key="PROJ", dry_run=true)
POST /api/jira/PROJ?dry_run=true
```

Real submissions return a per-story `results` list with `success`, `failure`, or `skipped` status. A single failed story does not stop later stories from being attempted. Successful real submissions still include the legacy `created_issues` list and `count` for existing callers.

## Notes

- `capture_requirement` now calls `text-embedding-3-small` for dedup and `gpt-4o` for confidence scoring on every requirement capture. Both calls are mocked in tests and lazy in voice mode.
- `generate_user_stories` uses Pydantic structured output via `client.beta.chat.completions.parse` with a JSON-schema fallback for compatible OpenAI client versions.
- Jira submission uses stdlib `urllib` only — no `requests` or `httpx` required in the core app.
- Sessions persist under `sessions/<project_key>/`. Flat-path sessions from earlier versions are still loaded as a fallback.
- Cost rates default to public OpenAI pricing but are fully configurable via env vars — update them when rates change.
- No credentials are hardcoded. All secrets are read from `.env` via `python-dotenv`.
